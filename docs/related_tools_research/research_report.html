FMI/FMU Validation & Testing Tools

FMPy
FMI · Python · Modelica Association

The de-facto standard Python library for simulating and validating FMUs. Offers a GUI, CLI, and API. Supports FMI 1.0, 2.0, 3.0. Powers the official fmi-standard.org web validator. Produces CSV time-series results suitable for regression comparison.

time-series CI-ready FMI 3.0 validation cross-check
Reference-FMUs + fmusim
FMI · C · Modelica Association

Official Modelica Association reference FMUs (BouncingBall, Clocks, etc.) plus the fmusim CLI simulator. Tests correct call-sequence compliance, event handling, and FMI state machine; outputs CSV. Indispensable for FMI implementors.

time-series event-detection FMI 2/3 official
FMI Cross-Check
FMI · Python · Modelica Association

Industry-wide cross-tool FMU interoperability validation suite. 1530+ FMUs tested across tool pairings. Defines reference CSV trajectories (*_ref.csv) and tolerance options (*_ref.opt) as a standard format for trajectory comparison. Currently being reworked.

time-series cross-simulator ref-trajectories
FMUComplianceChecker
FMI · C · Modelica-Tools (archived)

Historical FMU 1.0/2.0 compliance checker from QTronic/Synopsys. Now discontinued; FMPy and Reference-FMUs are the recommended replacements. Still useful for understanding the validation paradigm. Supported CSV input stimuli, log levels, and event checking.

FMI 1/2 only archived
libcosim (OSP)
FMI · C++ · DNV / Open Simulation Platform

Production-grade C++ co-simulation engine for FMI 1.0/2.0 used in maritime industry digital twins. Includes scenario runner, observer/manipulator interfaces, and CSV output. Python wrapper (libcosimpy) uses pytest. Fixed-step master algorithm with configurable step size.

time-series pytest SSP/OSP co-sim
libcosimpy
FMI · Python · Open Simulation Platform

Python wrapper for libcosim enabling pytest-based testing of FMU co-simulations. Tests can be structured as standard pytest fixtures; supports ECCO adaptive stepping algorithm for energy-conservation co-simulation. FMUs added manually or via SSP config files.

pytest time-series FMI 2.0

CI-ready schema-check FMI 3.0

Modelica · Python · OSMC

Official OSMC nightly regression test suite for Modelica libraries. Simulates models using OMC, stores results in SQLite, compares against reference MAT/CSV files with configurable tolerances. Reports published at libraries.openmodelica.org. Covers C++, FMI, daeMode runtimes.

Modelica · GitHub Actions · OSMC

GitHub Action that wraps OpenModelicaLibraryTesting for use in any Modelica library's CI pipeline. Installs OMC, runs simulations, compares against reference files, publishes HTML reports to GitHub Pages. Exemplary CI integration; accepts reference-files-dir parameter.

Modelica · Python/pytest · Open-source

Lightweight Python regression testing framework for Modelica libraries (Modelica 2023 paper). Uses .mos file interface so it works with any compliant Modelica tool — not locked to OMC. Supports pluggable comparison metrics. Integrates naturally with pytest and GitHub Actions.

pytest CSV comparison pluggable metrics
csv-compare
Modelica · C# · Modelica-Tools

Dedicated tool for comparing simulation result CSV files with adjustable absolute/relative tolerances. Used extensively in Modelica library development. Supports recursive directory comparison (batch mode). Published by the modelica-tools org; actively maintained.

time-series tolerance batch mode
OMSimulator
FMI/SSP · C++ · OSMC

FMI/SSP-based co-simulation environment integrated into OMEdit and OMC. Can be invoked as a testing backend in OpenModelicaLibraryTesting via the --fmisimulator flag. Enables cross-validation between native OMC results and FMU-based simulation results.

Julia · SciML

Core testing engine for the entire SciML ecosystem. Provides convergence testing, work-precision diagrams, solver shootouts, and error approximation. test_convergence() verifies numerical order of ODE/SDE solvers. Used in every SciML CI pipeline.

CI-ready convergence work-precision SDE support
SciMLBenchmarks.jl
Julia · SciML

Comprehensive benchmark suite for ODE, SDE, DAE solvers covering Julia, Python, MATLAB, and R. Work-precision diagrams built on low-tolerance reference solutions. Runs on self-hosted GitHub Actions with 32-core EPYC for reproducible hardware benchmarks. Published at benchmarks.sciml.ai.

GitHub Actions cross-lang reference-solutions DAE/SDE
DiffEqProblemLibrary.jl
Julia · SciML

Library of pre-made ODE, SDE, DDE, DAE test problems with known/analytic solutions for benchmarking and regression testing. Includes classical stiff problems (Robertson, Van der Pol), non-stiff problems, and biological models. Enables standard benchmark suite reproducibility.

benchmark suite analytic solutions stiff/non-stiff

Julia · SciML

Acausal symbolic modeling framework. Testing typically uses Julia's Test.jl with @test sol[x, end] ≈ ref atol=1e-6 patterns. Supports DAE index reduction, automatic sparsity detection, and structural analysis — all of which are testable properties.

Test.jl DAE symbolic

Julia-native FMI integration. FMI.jl loads/simulates FMUs seamlessly inside Julia. FMIFlux.jl enables NeuralFMU training — embedding FMUs inside neural networks. Testing can compare FMU trajectories against native Julia models using standard Test.jl tolerance assertions.

Python-Based General Testing Harnesses
5 approaches identified
FMPy + pytest pattern
Python · pytest · Community

The most common community pattern: wrap simulate_fmu() in a pytest test, compare result arrays using numpy.testing.assert_allclose() with rtol/atol. Golden CSV trajectories stored in tests/reference/ and loaded as numpy arrays for comparison.

pytest numpy comparison golden files
PyFMI
Python · LGPL · Modelon / OSMC

Mature Python package for FMU simulation (Cython-based, wraps FMIL). Supports FMI 1.0/2.0 for ME and CS. Trajectory results accessible as structured arrays; integrates with scipy for event detection. Widely used in academic and industrial workflows.

time-series FMI 1/2 Cython scipy
pytest-regressions (simulation)
Python · pytest ecosystem · Generic

General-purpose regression testing framework with data_regression (JSON/YAML), num_regression (numpy arrays), and dataframe_regression fixtures. num_regression.check() stores golden arrays and compares with configurable tolerances — directly applicable to simulation time-series outputs.

pytest numpy arrays golden files auto-update
PySimulator
Python · GUI · DLR / OpenModelica

Simulation and analysis environment with plugin-based simulator support (OpenModelica, Dymola, FMPy). Historical significance: its regression plugin was the first systematic cross-tool Modelica comparison framework. Now mostly superseded by MoPyRegtest and FMPy but still referenced in literature.

cross-tool GUI plugin-based
diffeqpy / diffeqr
Python/R → Julia · SciML

Bridges SciML's Julia solvers to Python and R. Enables Python test scripts to drive Julia ODE/SDE solvers, enabling cross-solver regression testing: compare a Python scipy result against a Julia DiffEq result for the same problem at matched tolerances.

Python↔Julia ODE/SDE pytest-compatible

Pros / Cons Analysis

In-depth analysis of each major tool or approach, covering practical strengths and documented weaknesses from the developer community.

FMPy — Primary FMU simulation & validation library
ACTIVE

Strengths

  • Official Modelica Association backing; most widely-used FMU Python tool
  • Supports FMI 1.0, 2.0, and 3.0 natively including SSP
  • Built-in validate_fmu() for structural/schema validation
  • GUI, CLI, and full Python API — zero lock-in
  • Powers fmi-standard.org online validator; battle-tested
  • Cross-check validation script built-in (fmpy.cross_check)
  • Easy pytest integration: results as structured numpy arrays

Weaknesses

  • No built-in golden-file management; requires manual numpy/CSV comparison
  • Event detection testing requires custom logic; no built-in event assertion
  • No stochastic simulation or ensemble support
  • FMI 3.0 support is newer and less battle-tested than 2.0
  • No native cross-simulator validation (comparing FMU vs. native model)
OpenModelicaLibraryTesting + GitHub Action
ACTIVE

Strengths

  • Fully automated nightly CI; covers 50+ Modelica libraries
  • Reference MAT/CSV files with configurable tolerance via JSON config
  • HTML report generation + GitHub Pages deployment built-in
  • Covers multiple runtimes: C++, FMI, daeMode, old frontend
  • GitHub Action packaging makes integration trivial for library authors
  • SQLite result storage enables historical trend analysis

Weaknesses

  • OMC install is heavy (~1GB); CI compute cost is significant
  • Limited to OpenModelica as simulator (despite .mos interface hints)
  • Reference file generation requires running OMC — bootstrapping problem
  • No stochastic simulation support
  • Test discovery depends on Modelica experiment annotations (StopTime)
  • Cross-simulator validation requires manual OMSimulator setup
MoPyRegtest
MODERATE

Strengths

  • Tool-agnostic via .mos scripting interface (OpenModelica, Dymola, etc.)
  • Pluggable comparison metrics — easy to add custom error measures
  • Native pytest integration; tests look like standard Python unit tests
  • Lightweight — minimal dependencies, easy to install
  • Formally published (Modelica 2023 conference paper)

Weaknesses

  • Small community; limited real-world adoption reports
  • No built-in parallelism for large test suites
  • No HTML report generation; results are pytest output only
  • No event detection, stochastic, or cross-simulator support
  • Requires simulator to support .mos scripting (not all do)
DiffEqDevTools.jl + SciMLBenchmarks.jl
ACTIVE

Strengths

  • Purpose-built for numerical solver correctness verification
  • Work-precision diagrams reveal accuracy vs. compute trade-offs
  • Convergence testing verifies theoretical solver order empirically
  • Covers ODE, SDE, DDE, DAE — stochastic support is a differentiator
  • Cross-language benchmarking (Julia, Python, MATLAB, R) via SciMLBenchmarks
  • Deeply integrated with SciML CI; every solver PR runs convergence tests
  • Reference solutions at ultra-low tolerance to bootstrap comparisons

Weaknesses

  • Julia-ecosystem-only; difficult to apply to non-Julia models
  • No FMI/FMU support (FMI.jl bridges this but adds complexity)
  • Self-hosted CI runners needed for reproducible benchmarks (expensive)
  • Work-precision diagrams don't directly map to regression testing of model outputs
  • No golden-file based trajectory comparison for end-user models
libcosim / cosim-cli (OSP)
ACTIVE

Strengths

  • Production-grade co-simulation engine; actively used in maritime industry
  • Scenario runner enables systematic input sweep testing
  • Observer/manipulator pattern allows variable injection mid-simulation
  • SSP standard support for multi-FMU system-level testing
  • Python wrapper with pytest; ECCO adaptive stepping for energy conservation

Weaknesses

  • Complex setup: requires Conan, CMake, C++ build chain
  • FMI 3.0 support not yet available (2.0 only as of early 2026)
  • No built-in tolerance-based trajectory comparison — must write your own
  • Maritime/engineering domain focus; tooling UX reflects that
  • Small developer community outside Nordic/maritime industry

Gaps in the Ecosystem

Critical missing capabilities, community pain points, and areas dominated by proprietary tooling. These represent high-value opportunities for open-source development.

🔴 Critical Missing Capabilities

No Unified, Standardized Regression Framework

There is no single framework that covers the full workflow: simulate → store golden trajectory → compare with tolerance → report diff → update golden on intent. Each ecosystem has partial solutions (csv-compare, MoPyRegtest, DiffEqDevTools) but no standard that works across FMI, Modelica, and ODE solver ecosystems simultaneously. The FMI Cross-Check provides a standard format (*_ref.csv / *_ref.opt) but it is only tool-to-tool, not general-purpose.

No Standardized Event Detection Testing

Hybrid systems (bouncing balls, switching controllers, thermostat models) require verifying that discrete events occur at the correct times. No open-source tool provides a declarative assertion language for event timing, like assert event(x_crosses_zero) at t ≈ 0.5s ± 1ms. FMPy and PyFMI detect events but provide no structured test assertion framework around them.

No Stochastic Simulation Regression in FMI/Modelica Ecosystems

SciML/DiffEqDevTools handles SDE convergence testing in Julia, but the FMI and Modelica ecosystems have zero standardized support for stochastic simulation regression. Comparing stochastic trajectories requires statistical tests (KS-test, confidence interval checking) rather than point-wise tolerances, and none of the FMI/Modelica tools support this.

Weak Cross-Simulator Validation Tooling

While FMI.jl enables FMU↔Julia comparison, and the FMI Cross-Check enables tool-to-tool comparison, there is no general-purpose framework for validating that a model gives consistent results across simulators (e.g., FMPy vs. PyFMI vs. libcosimpy for the same FMU; or OpenModelica vs. Dymola for the same Modelica model). Cross-tool discrepancies beyond expected numerical tolerance are a well-known industry problem.

🟡 Reported Pain Points

Golden Reference File Management

No simulation-specific tool provides automatic golden file update workflows (analogous to pytest --update-goldens or Rust insta snapshots). Developers must manually regenerate reference CSVs and commit them, with no tooling to diff, review, or selectively update changed trajectories. This is particularly painful for large Modelica libraries with hundreds of models.

CI/CD Compute Cost for Simulation Tests

Heavy simulators (OMC, Dymola) make CI expensive. The OM library testing documentation explicitly notes that installing OMC in GitHub Actions requires "significant computational resources and valuable usage time." Long simulation runtimes conflict with fast feedback CI expectations. Solutions like Docker caching and pre-built OMC images help but are not standardized.

Numerical Drift and Platform-Specific Results

The same model compiled on different OS/compiler combinations produces numerically different trajectories even with identical solver settings. This makes bit-exact regression testing impossible and requires tolerance-based comparison — but choosing appropriate tolerances is non-trivial and not well-documented. The FMI Cross-Check's ref.opt format is the closest to a standard approach.

🟣 Proprietary-Dominated Areas

Dymola Regression Testing

Dymola (Dassault Systèmes) is the most widely used Modelica tool in automotive/aerospace industries. Its regression testing is built into the Dymola environment and is proprietary. Open-source alternatives that can process Dymola's result files (MAT format) exist (buildingspy, OMPython reading MAT) but a Dymola-compatible open-source CI harness does not.

Simulink / MATLAB Ecosystem

MATLAB/Simulink testing relies on proprietary Simulink Test and Model Advisor. While FMIKit-Simulink enables FMU import/export, regression testing of Simulink models in open-source CI pipelines is essentially unsupported. Simulix (GPL3, generates FMUs from Simulink via C-API) is the closest open bridge, but it is experimental and limited.

Physics-Based CAE Simulation Testing

Large-scale FEA/CFD simulation regression (OpenFOAM, FEniCS, ANSYS) is a largely separate ecosystem. These simulations produce spatial-temporal field data rather than time-series scalars, requiring different comparison infrastructure. No tool bridges this domain with the FMI/Modelica regression paradigm.

Recommended Architectures / Patterns

Battle-tested patterns for structuring simulation regression tests. Each includes a concrete code example and CI configuration. Links to real repository implementations are provided where available.

PATTERN 01 · MOST COMMON

Golden Trajectory Comparison with Tolerances (FMPy + pytest + numpy)

The most widely adopted pattern. Simulate an FMU, store CSV trajectories as golden references in version control, and compare future results within tolerance bands. Works for any FMU via FMPy. Uses numpy.testing.assert_allclose for vectorized tolerance comparison.

📁 Real example: modelica/fmi-cross-check uses this exact pattern with *_ref.csv + *_ref.opt files
# tests/test_bouncing_ball.py import numpy as np import pytest from fmpy import simulate_fmu, read_model_description GOLDEN_CSV = "tests/reference/BouncingBall_ref.csv" FMU_PATH = "models/BouncingBall.fmu" def test_bouncing_ball_trajectory(): result = simulate_fmu( FMU_PATH, stop_time=3.0, output_interval=0.01, output_variables=['h', 'v'] ) reference = np.genfromtxt(GOLDEN_CSV, delimiter=',', names=True) np.testing.assert_allclose( result['h'], reference['h'], rtol=1e-4, atol=1e-6, err_msg="Height trajectory deviates from reference" ) def test_fmu_validates(): from fmpy.validation import validate_fmu problems = validate_fmu(FMU_PATH) assert problems == [], f"FMU validation failed: {problems}"
# .github/workflows/fmu-regression.yml name: FMU Regression Tests on: [push, pull_request] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: {python-version: '3.11'} - run: pip install fmpy pytest numpy - run: pytest tests/ -v --tb=short

PATTERN 03 · JULIA/SCIML

SciML Convergence Testing + Snapshot Regression

Standard pattern in the SciML ecosystem. Uses DiffEqDevTools.jl for convergence-order verification, and Julia's native Test.jl for point-in-time trajectory regression. The DiffEqProblemLibrary.jl provides standard test problems.

# test/test_van_der_pol.jl using Test, OrdinaryDiffEq, DiffEqDevTools, DiffEqProblemLibrary ODEProblemLibrary.importodeproblems() # Convergence test: verify solver achieves theoretical order @testset "Van der Pol convergence order" begin prob = ODEProblemLibrary.prob_ode_vanderpol abstols = 1 ./ 10 .^ (4:7) reltols = 1 ./ 10 .^ (4:7) sim = test_convergence(abstols, reltols, prob, Tsit5()) @test sim.𝒪est[:final] ≈ 4 atol=0.3 # Tsit5 is order 4 end # Trajectory regression: snapshot at key time points @testset "Van der Pol trajectory regression" begin prob = ODEProblem((u,p,t) -> [u[2]; ((1-u[1]^2)*u[2]-u[1])], [2.0, 0.0], (0.0, 10.0)) sol = solve(prob, Tsit5(), abstol=1e-10, reltol=1e-10) # Compare endpoint against known reference @test sol[1, end] ≈ REFERENCE_X1_ENDPOINT rtol=1e-6 @test sol[2, end] ≈ REFERENCE_X2_ENDPOINT rtol=1e-6 end
PATTERN 04 · CROSS-SIMULATOR

FMU vs. Native Model Cross-Validation (FMI.jl + ModelingToolkit.jl)

Using Julia's FMI.jl to simulate an FMU alongside a native MTK.jl model of the same system. Validates that the FMU export preserves model semantics by comparing time-series trajectories within solver-order-appropriate tolerances.

# Cross-validate Modelica FMU against native Julia ModelingToolkit model using FMI, ModelingToolkit, OrdinaryDiffEq, Test # --- Simulate the FMU --- fmu = loadFMU("BouncingBall.fmu") fmu_solution = simulate(fmu, (0.0, 3.0)) # --- Native Julia model (equivalent physics) --- @variables t h(t) v(t) D = Differential(t) eqs = [D(h) ~ v; D(v) ~ -9.81] # simplified sys = ODESystem(eqs, t) native_sol = solve(ODEProblem(sys, [h=>1.0, v=>0.0], (0,3.0))) # --- Cross-compare at matched time points --- t_compare = 0.0:0.1:3.0 @test fmu_solution.values.saveval ≈ native_sol(t_compare)[1,:] rtol=1e-3 unloadFMU(fmu)
PATTERN 05 · ADVANCED

Hybrid Python Testing Harness: pytest-regressions + FMPy + Scenario Parameterization

A production-grade pattern combining pytest-regressions for golden-file management, FMPy for simulation, and pytest.mark.parametrize for scenario coverage. The num_regression fixture handles the golden-file update workflow automatically.

# tests/test_heat_pump.py — pytest-regressions + FMPy pattern import pytest import numpy as np from fmpy import simulate_fmu from pytest_regressions import num_regression # noqa: F401 (fixture) SCENARIOS = [ ("nominal", {"T_set": 293.15, "T_ambient": 275.0}), ("cold_day", {"T_set": 293.15, "T_ambient": 255.0}), ("hot_day", {"T_set": 293.15, "T_ambient": 300.0}), ] @pytest.mark.parametrize("name,params", SCENARIOS) def test_heat_pump_scenario(name, params, num_regression): result = simulate_fmu( "HeatPump.fmu", start_values=params, stop_time=3600, output_interval=60, output_variables=['T_supply', 'P_compressor', 'COP'] ) # First run: creates reference CSV; subsequent runs: compares with tolerance num_regression.check( {'T_supply': result['T_supply'], 'P_compressor': result['P_compressor'], 'COP': result['COP']}, default_tolerance={"rtol": 1e-4, "atol": 0}, basename=f"heat_pump_{name}" ) # Update goldens: pytest --force-regen
⚠ When using pytest-regressions, commit generated golden files to version control. Use --force-regen only on intentional model changes. Include golden-file diffs in code review.
PATTERN 06 · CI/CD PIPELINE

Full CI-Driven Simulation Verification Pipeline

A complete GitHub Actions workflow strategy for simulation-heavy projects. Separates fast unit tests (model validation, schema checks) from slow integration tests (full simulation + trajectory comparison) using job dependencies and caching.

# .github/workflows/simulation-ci.yml — Multi-stage pipeline name: Simulation Regression CI on: [push, pull_request] jobs: # Stage 1: Fast checks (no simulation needed) validate: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: pip install fmpy - name: Validate all FMUs run: | python -c " from fmpy.validation import validate_fmu import glob, sys errors = [] for fmu in glob.glob('models/**/*.fmu', recursive=True): problems = validate_fmu(fmu) if problems: errors.extend([f'{fmu}: {p}' for p in problems]) if errors: sys.exit('\n'.join(errors)) " # Stage 2: Simulation regression (requires golden files) regression: needs: validate runs-on: ubuntu-latest strategy: matrix: test-group: [fmu-tests, modelica-tests, system-tests] steps: - uses: actions/checkout@v4 with: {lfs: true} # Golden files in Git LFS - uses: actions/setup-python@v5 with: {python-version: '3.11'} - uses: actions/cache@v4 with: path: ~/.cache/fmpy key: fmpy-${{ hashFiles('models/**/*.fmu') }} - run: pip install fmpy pytest numpy pytest-regressions - run: pytest tests/${{ matrix.test-group }}/ -v --tb=long env: REGRESSION_RTOL: "1e-4" REGRESSION_ATOL: "1e-8" # Stage 3: Cross-simulator validation (scheduled, not on every PR) cross-validate: needs: regression if: github.event_name == 'schedule' runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: docker run --rm -v $PWD:/work myorg/om-fmpy-compare
Key architectural principle: Separate structural validation (FMU schema, XML, exported variables) from numerical regression (trajectory comparison) from cross-simulator validation into distinct CI stages. Only the first is fast enough for every commit; the rest can be scheduled or triggered on PR merges.
